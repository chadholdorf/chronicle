# Chronicle — Developer Context

## Project purpose

Chronicle is an open source Google Reader revival. It's a local-first RSS reader with AI features powered by the Claude API. The primary audience is a senior product leader using this as a portfolio project, so code quality and UX polish matter as much as functionality.

## Stack

- **Next.js 14** with App Router and TypeScript (strict mode)
- **SQLite** via **Drizzle ORM** — `better-sqlite3` driver, database at `./chronicle.db`
- **Tailwind CSS** — CSS custom properties defined in `globals.css` for the design system
- **Claude API** (`claude-sonnet-4-20250514`) via `@anthropic-ai/sdk`
- **rss-parser** for feed fetching and parsing

## Database schema

Three tables, defined in `db/schema.ts`:

- **feeds** — Feed subscriptions. `unread_count` and `item_count` are denormalized and kept in sync after every write.
- **items** — Individual feed articles. `summary` column is populated lazily by Claude and cached so we never call the API twice for the same article.
- **digests** — Saved daily digests generated by Claude.

Schema is initialized via raw SQL in `app/lib/db.ts` (`initSchema`). No migration runner — the schema is idempotent CREATE IF NOT EXISTS. If you add columns, you need to handle that manually or reset the DB.

## Key architectural decisions

### SQLite initialization pattern
`app/lib/db.ts` exports a `db` proxy that lazily initializes the connection on first use. This avoids creating a connection during the Next.js module import phase, which causes issues with edge environments. The Drizzle client wraps `better-sqlite3` directly (no connection pool needed — SQLite is single-writer).

### No migration runner
Drizzle Kit is installed but we're not using its migration workflow. Instead, `initSchema()` runs `CREATE TABLE IF NOT EXISTS` on startup. This is deliberate for a local-first tool — there's no deployment pipeline to coordinate. If you want to add columns, add them to both the schema file and the SQL in `initSchema`.

### Unread counts are denormalized
`feeds.unread_count` is updated after every item read/mark-all-read operation. This is a deliberate tradeoff: badge counts need to be fast and the mutation surface is small.

### AI summaries are cached
`items.summary` is populated on first call to `/api/summarize` and returned from the DB on subsequent calls. The API route checks for an existing summary before calling Claude.

## How feed refresh works

1. On app load, `page.tsx` calls `POST /api/refresh` with no body — this refreshes any feed not updated in the last 30 minutes.
2. "Refresh All" button calls the same endpoint with `{ force: true }` — this refreshes every feed regardless of age.
3. `refreshFeed()` in `app/lib/rss.ts` fetches the feed URL, builds item records, and inserts with `onConflictDoNothing()` (deduplication by `url` unique constraint).
4. After insert, it updates `feeds.last_fetched_at` and calls `syncFeedCounts()`.

## How Claude API is called

Two functions in `app/lib/claude.ts`:

- `summarizeArticle(content, title)` — Article summary. System prompt enforces 3-sentence structure: what happened, why it matters, what to watch.
- `generateDailyDigest(items)` — Daily briefing. System prompt enforces 4-section structure. Explicitly forbids em dashes (they're a Claude tell).

Both use `claude-sonnet-4-20250514`. Neither uses streaming — responses are short enough that full response latency is acceptable.

## Local dev setup

```bash
npm install
cp .env.example .env.local   # add ANTHROPIC_API_KEY
npm run dev
```

The SQLite DB (`chronicle.db`) is created on first request. To reset: `rm chronicle.db`.

AI features degrade gracefully — the app works without `ANTHROPIC_API_KEY`, but the Summarize button and Daily Digest will return a 503.

## What to build next

These are the highest-value missing features, roughly in priority order:

1. **Keyboard shortcuts** — `j/k` to navigate items, `Enter` to open, `s` to star, `r` to mark read, `o` to open original. Google Reader muscle memory.
2. **OPML import/export** — Import your existing subscriptions from Feedly or other readers; export for backup.
3. **Mobile responsive layout** — The three-panel layout needs to collapse to a single-panel drawer nav on small screens.
4. **Starred items export** — Download starred items as JSON or Markdown.
5. **Feed categories/folders** — Group feeds in the sidebar (needs a new `categories` table and a `feed.category_id` FK).
6. **Full-text search** — SQLite FTS5 extension would make this straightforward.

## Design system

CSS custom properties are defined in `globals.css`:

```
--accent: #d97706  (warm amber — RSS orange nod)
--bg-secondary: #f7f7f5
--border: #e5e4e0
```

The reading pane uses Georgia serif for article content (`.reading-content` class). Everything else uses Inter.

## File structure

```
app/
  api/
    feeds/          GET (list), POST (add)
    feeds/[id]/     DELETE
    items/          GET (list with filters)
    items/[id]/     PATCH (isRead, isStarred)
    items/mark-read/ POST (bulk mark read)
    refresh/        POST (refresh stale or all feeds)
    summarize/      POST (Claude summary, cached)
    digest/         GET (latest digest), POST (generate new)
  components/
    FeedList.tsx    Left panel
    ItemList.tsx    Center panel
    ReadingPane.tsx Right panel
    AddFeedModal.tsx
    DigestModal.tsx
  lib/
    db.ts           Drizzle client + schema init
    rss.ts          Feed fetching, parsing, refresh
    claude.ts       Claude API wrapper
    time.ts         formatDistanceToNow utility
  page.tsx          Root client component, all state lives here
db/
  schema.ts         Drizzle schema types
```
